# 大语言模型（LLM）

大语言模型（Large Language Model，LLM）是一类基于深度学习的自然语言处理模型，通过在海量文本上进行预训练，学习语言的统计规律、世界知识与推理模式，从而能够完成文本生成、理解、翻译、摘要、问答等多种任务。

## 核心概念

### 什么是 LLM

LLM 通常指参数量达到数十亿乃至数千亿级别的语言模型。与早期基于规则或浅层统计的方法不同，LLM 以 **自回归** 或 **掩码语言建模** 为目标，在 Transformer 架构上规模化训练，具备较强的泛化能力与涌现能力（Emergent Abilities）。

### 关键术语

| 术语 | 说明 |
|------|------|
| **Token** | 模型处理的最小文本单元，可能是字、词或子词（BPE/SentencePiece） |
| **上下文窗口（Context Window）** | 单次推理可容纳的最大 Token 数，决定模型能“看到”多长的历史 |
| **温度（Temperature）** | 控制采样随机性，越高输出越多样，越低越确定 |
| **Top-p / Top-k** | 限制采样候选集，在多样性与质量之间权衡 |
| **幻觉（Hallucination）** | 模型生成看似合理但事实错误的内容 |
| **对齐（Alignment）** | 使模型输出符合人类价值观与指令意图的训练过程 |

## 技术架构

### Transformer 基础

现代 LLM 几乎均基于 **Transformer**（Vaswani et al., 2017）：

- **自注意力（Self-Attention）**：让每个 Token 关注序列中其他位置，捕获长距离依赖
- **多头注意力（Multi-Head Attention）**：并行学习多种关系模式
- **位置编码**：弥补注意力机制本身不包含顺序信息的问题（绝对/相对/RoPE 等）
- **前馈网络（FFN）**：对每个位置做非线性变换

常见变体包括 **仅解码器（Decoder-only）**、**编码器-解码器（Encoder-Decoder）** 等；GPT 系列、LLaMA、Qwen 等主流开源模型多采用 Decoder-only。

### 模型规模与能力

参数量、训练数据量与算力通常遵循 **缩放定律（Scaling Laws）**：在一定范围内，模型越大、数据越多，下游任务表现越好，并可能出现小模型不具备的能力（如复杂推理、代码生成）。

```
预训练数据 → Tokenizer → Transformer 堆叠 → 语言建模头 → 下一个 Token 预测
```

## 训练流程

LLM 的完整生命周期通常分为以下阶段：

### 1. 预训练（Pre-training）

在大规模无标注语料（网页、书籍、代码等）上进行 **自监督学习**，目标多为：

- **因果语言建模（CLM）**：根据前文预测下一个 Token（GPT 类）
- **掩码语言建模（MLM）**：预测被遮盖的 Token（BERT 类，多用于理解任务）

预训练使模型掌握语法、常识与一定推理能力，但未必善于遵循用户指令。

### 2. 监督微调（SFT）

使用高质量的「指令 - 回答」对进行有监督微调，使模型学会 **按指令完成任务**（对话、写作、分析等）。

### 3. 人类反馈强化学习（RLHF / RLAIF）

通过人类或 AI 对多个回答排序，训练 **奖励模型（Reward Model）**，再用 PPO、DPO、GRPO 等方法优化策略，使输出更安全、有用、诚实。

### 4. 持续迭代

包括领域微调、量化部署、检索增强（RAG）、工具调用（Function Calling / Agent）等工程化扩展。

::: tip 训练与推理的区别
训练需要反向传播与大量 GPU 显存；推理（Inference）仅做前向计算，关注延迟、吞吐与成本，常采用量化（INT8/INT4）、KV Cache、批处理等优化。
:::

## 典型应用场景

- **对话与助手**：智能客服、个人助理、多轮问答
- **内容创作**：文章、营销文案、剧本、邮件
- **代码**：补全、解释、重构、单元测试生成（Copilot、Cursor 等）
- **知识工作**：摘要、翻译、信息抽取、报告分析
- **RAG 系统**：结合向量数据库，用私有文档增强回答准确性
- **Agent**：结合规划、记忆与工具调用，完成多步骤复杂任务

## 主流模型概览

| 类型 | 代表 | 特点 |
|------|------|------|
| 闭源 API | GPT-4o、Claude、Gemini | 能力强、易用，依赖云端 API |
| 开源通用 | LLaMA、Qwen、Mistral、DeepSeek | 可本地部署、可微调 |
| 代码专用 | CodeLlama、StarCoder、DeepSeek-Coder | 代码理解与生成优化 |
| 多模态 | GPT-4V、LLaVA、Qwen-VL | 同时处理文本与图像 |

选型时需综合考虑：**能力、成本、延迟、隐私合规、是否支持中文与领域知识**。

## 使用 LLM 的最佳实践

### 提示工程（Prompt Engineering）

- 角色与任务描述清晰（System / User 分工）
- 提供示例（Few-shot）或输出格式约束（JSON、Markdown）
- 复杂任务拆分为多步（Chain-of-Thought、ReAct）
- 对长文档使用分块、摘要或 RAG，避免超出上下文窗口

### 控制输出质量

```text
低温度（0～0.3）  → 事实性、格式稳定的任务
中温度（0.5～0.8）→ 通用对话与创作
高温度（>1）      → 创意发散，但幻觉风险上升
```

### 安全与合规

- 对用户输入做过滤，防止提示注入（Prompt Injection）
- 敏感数据勿直接送入不可信第三方 API
- 关键业务场景需人工审核或与传统系统交叉验证

## 局限与挑战

- **知识截止**：预训练数据有时间边界，实时信息需 RAG 或工具
- **幻觉**：可能编造引用、数据或逻辑
- **推理成本**：长上下文与高并发带来算力与费用压力
- **偏见与安全**：训练数据中的偏见可能被放大
- **可解释性**：难以精确说明某一回答的内部依据
