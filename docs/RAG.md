# 检索增强生成（RAG）

检索增强生成（Retrieval-Augmented Generation，RAG）是一种将 **外部知识检索** 与 **大语言模型生成** 相结合的技术架构。模型在回答用户问题前，先从知识库中检索相关文档片段，再基于检索结果生成回答，从而缓解幻觉、弥补知识截止，并支持私有数据问答。

## 为什么需要 RAG

纯 LLM 存在以下局限：

| 问题 | RAG 的应对方式 |
|------|----------------|
| **知识截止** | 从实时或私有知识库检索最新信息 |
| **幻觉** | 回答有据可查，可引用来源片段 |
| **领域知识不足** | 注入企业文档、手册、代码库等专业内容 |
| **上下文有限** | 只将最相关的片段送入模型，而非整库全文 |
| **数据隐私** | 知识留在本地向量库，无需把全部数据写入模型权重 |

RAG 并不替代模型微调，而是与 **Prompt 工程、微调、Agent** 等方案互补，常用于企业知识库、客服问答、代码助手等场景。

## 工作原理

典型 RAG 流水线分为 **离线索引** 与 **在线推理** 两阶段：

```
【离线】文档 → 切分 Chunk → Embedding → 写入向量数据库

【在线】用户问题 → Query Embedding → 相似度检索 → Top-K 片段
                                              ↓
                                    拼接 Prompt → LLM 生成 → 返回答案
```

### 核心组件

| 组件 | 作用 |
|------|------|
| **文档加载器（Loader）** | 从 PDF、网页、数据库等来源读取原始文本 |
| **文本切分（Chunking）** | 将长文档拆为适合检索与上下文窗口的片段 |
| **嵌入模型（Embedding）** | 将文本映射为稠密向量，用于语义相似度计算 |
| **向量数据库（Vector Store）** | 存储向量并支持 ANN 近似最近邻检索 |
| **检索器（Retriever）** | 根据查询返回 Top-K 相关 Chunk |
| **重排序（Reranker）** | 对初检结果精排，提升相关性（可选） |
| **生成器（LLM）** | 结合检索上下文与用户问题生成最终回答 |

::: tip 与微调的对比
- **RAG**：知识可热更新，适合事实性、文档型问答，实施快
- **微调**：改变模型行为与风格，适合固定任务与领域表达，知识更新成本高
- 实践中常 **RAG + 微调** 组合使用
:::

## 文档切分策略

切分质量直接影响检索效果，常见策略如下：

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **固定长度** | 按字符/Token 数切分，可重叠（Overlap） | 通用文本 |
| **递归切分** | 按标题、段落、句子层级递归拆分 | 结构化文档 |
| **语义切分** | 按语义边界或嵌入相似度断句 | 长文、论述类 |
| **元数据增强** | 为 Chunk 附加标题、页码、章节路径 | 需要引用溯源 |

推荐参数参考（需按模型上下文调整）：

- Chunk 大小：**300～800 Token**
- Overlap：**10%～20%**，避免关键信息被截断在边界

## 检索与增强

### 向量检索

1. 将用户问题编码为查询向量
2. 在向量库中计算余弦相似度或内积
3. 返回 Top-K（通常 **3～10**）个最相关片段

### 混合检索（Hybrid Search）

结合 **稠密向量检索** 与 **稀疏检索（BM25/关键词）**，兼顾语义理解与精确词匹配，在专有名词、编号、代码等场景表现更好。

### 重排序（Reranking）

使用 Cross-Encoder 等模型对初检 Top-N（如 50）结果二次打分，取 Top-K 送入 LLM，可显著提升准确率，代价是额外延迟。

### 查询增强

在检索前对问题做变换，例如：

- **HyDE**：先让 LLM 生成假设性答案，再用其做检索
- **多查询**：将原问题改写为多个检索 query 并合并结果
- **子问题分解**：复杂问题拆成多个子查询分别检索

## RAG 架构演进

| 类型 | 特点 |
|------|------|
| **Naive RAG** | 检索 → 拼接 → 生成，实现简单，易有噪声与冗余 |
| **Advanced RAG** | 增加预处理（清洗、元数据）、检索后压缩与重排 |
| **Modular RAG** | 模块化流水线，可插拔路由、多路检索、自我反思 |
| **Agentic RAG** | 由 Agent 决定何时检索、检索什么、是否多轮迭代 |

## Prompt 构建示例

将检索结果与用户问题组合后送入 LLM：

```text
你是一个专业的问答助手。请仅根据以下参考资料回答问题；
若资料中无法找到答案，请明确说明「根据现有资料无法回答」，不要编造。

【参考资料】
{context}

【用户问题】
{question}
```

::: warning 注意
- 控制 `context` 总长度，避免超出模型上下文窗口
- 明确约束模型 **不要脱离资料编造**
- 对敏感场景可增加引用标注（如 `[1][2]`）
:::

## 评估指标

| 维度 | 常见指标 |
|------|----------|
| **检索质量** | Recall@K、MRR、NDCG |
| **生成质量** | 人工评分、LLM-as-Judge、忠实度（Faithfulness） |
| **端到端** | 答案准确率、引用命中率、用户满意度 |

可使用 Ragas、TruLens 等框架做自动化评测，并结合真实业务问答集持续迭代。

## 最佳实践

1. **数据质量优先**：清洗乱码、去重、统一格式，劣质文档会污染检索
2. **小步验证**：先在小规模文档集上验证切分与检索，再扩展全库
3. **监控检索命中率**：记录「无结果」「低分检索」case，优化切分与 Embedding
4. **引用溯源**：返回答案时附带来源文档与页码，便于用户核实
5. **权限隔离**：多租户场景下在检索层做过滤，避免跨库泄露
6. **缓存与增量更新**：对稳定知识库做增量索引，避免全量重建

## 常见挑战

- **检索不准**：切分过粗/过细、Embedding 与领域不匹配
- **上下文噪声**：无关片段干扰生成，需 Rerank 或压缩
- **多跳推理**：单轮检索不足以回答复杂问题，需 Agent 或多轮检索
- **表格与图片**：纯文本 RAG 难以处理，需多模态解析或结构化抽取
- **延迟与成本**：Embedding、检索、Rerank、LLM 串联增加响应时间

## 与 LLM 的关系

RAG 建立在 LLM 之上：检索提供 **事实依据**，LLM 负责 **理解、整合与表达**。理解 [大语言模型](./llm) 的上下文窗口、幻觉与 Prompt 技巧，有助于更好地设计与调优 RAG 系统。
